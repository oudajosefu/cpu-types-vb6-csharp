The long Split — and how C# sidesteps it entirely

64-bit C: two philosophies, one headache · C#: just picks one and guarantees it

🐧 C on Unix / Linux / macOS

"We're on 64-bit hardware. long should match." Old code may break — deal with it.

C long = 64 bits
🪟 C on Windows

"Too many corporate apps depend on long=32. We invented long long for 64-bit math instead."

C long = 32 bits · long long = 64 bits
🟢 C# / .NET (all platforms)

The CLR spec simply declares: long = System.Int64 = 64 bits. Always. On every OS. The runtime enforces it — it's not up to the compiler or platform.

C# long = 64 bits everywhere ✓

VB6 & C# — types with no direct C integer equivalent
QUIRKY
VB6 Boolean
16 bits
≈ C short (but −1 = True, 0 = False)
True is −1 (all bits set), not 1. C's _Bool is 1 bit. COM defined VARIANT_TRUE as 0xFFFF. Never compare directly.
C# bool
8 bits (1 byte in memory)
≈ C _Bool / uint8_t
Logically 1 bit, but stored as 1 byte by the CLR. True = 1, False = 0. Unlike VB6 Boolean — no −1 surprises.
VB6 Single
32 bits
= C float (IEEE 754)
32-bit floating point. Direct match to C float. Not an integer type — fraction × exponent.
C# float / double
32 / 64 bits
= C float / double (IEEE 754)
float = System.Single (32-bit), double = System.Double (64-bit). Direct CLR-guaranteed matches to C's floating-point types.
VB6 ONLY
VB6 Currency
64 bits
≈ int64_t ÷ 10,000
A 64-bit integer scaled by 10,000 to give 4 decimal places. C has no equivalent — use int64_t with manual scaling. C# replaced this with decimal.
C# ONLY
C# decimal
128 bits
no C equivalent
System.Decimal: 96-bit integer + scale + sign. NOT IEEE 754. Avoids floating-point rounding for money math. C has nothing like it — would need a library.
VB6 ONLY
VB6 Date
64 bits
≈ C double (different epoch)
Stored as a double. Integer part = days since Dec 30, 1899. C uses time_t: seconds since Jan 1, 1970. Same bit width, completely different meaning.
VB6 ONLY
VB6 Variant
16 bytes
≈ tagged union (no C equiv)
2-byte type tag + 6 bytes reserved + 8 bytes data. Can hold any type at runtime. C# replaced this with object (reference) or generics.
VB6 String
variable (BSTR)
≈ wchar_t* (not null-terminated)
Length-prefixed UTF-16 (BSTR). C strings are null-terminated char arrays. C# string is a managed UTF-16 object. All three incompatible without marshalling.
C# string
variable (managed heap)
≈ VB6 String (UTF-16, not BSTR)
Immutable sequence of UTF-16 chars on the managed heap. NOT null-terminated, NOT a BSTR. Interop with C requires Marshal.StringToHGlobalAnsi etc.
C# object
pointer-sized reference
≈ void* (managed)
System.Object: root of the C# type hierarchy. A reference (pointer) to a heap-managed object. C has no GC — you'd use void* and manage memory manually.
C# nint / nuint
32 or 64 bits
= C intptr_t / size_t
Added in C# 9 / .NET 5. Platform-sized integers — 32-bit on 32-bit runtimes, 64-bit on 64-bit. Designed for native interop where pointer arithmetic is needed.

Notes: C# type sizes are guaranteed by the ECMA-334 / CLI spec, not by the platform — this is the key difference from C. VB6 was 32-bit only (~1998–2008). VBA 7.0 (Office 2010+) added LongLong and LongPtr for 64-bit hosts. C# decimal is 128-bit but not IEEE 754 — it's a high-precision base-10 type for financial math. VB6 Boolean True = −1 due to COM's VARIANT_TRUE = 0xFFFF. C99 fixed-width types: include <stdint.h> in C or <cstdint> in C++.
